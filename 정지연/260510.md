# ✔️ 왜 Main Thread는 죽지 않을까?

액티비티가 실행되면 아래와 같은 생명주기 메서드가 순서대로 호출된다.
`onCreate()`, `onStart()`, `onResume()`

이후에는 화면이 모두 그려지고 사용자 입력만 기다리는 상태가 된다.

그런데도 **Main Thread(UI Thread)는 종료되지 않는다.**
사용자가 화면을 터치하면 즉시 이벤트가 처리되고 UI가 반응한다.

어떻게 이게 가능한 것인가?????????
> 그 이유는 바로 **Looper** 때문!!

---


## 📌 핵심 개념

### ✅ Thread
> 실제 작업이 실행되는 단위

- 메인 스레드는 다음 역할을 담당한다.

* UI 그리기
* 터치 이벤트 처리
* 키보드 입력 처리
* Activity/Fragment 라이프사이클 호출


### ✅ Looper
> Looper는 스레드를 계속 살아있게 만드는 **무한 루프(loop)** 이다.

Looper는 아래 과정을 반복한다.

1. Message Queue에서 작업(Message/Runnable)을 꺼냄
2. 작업 실행
3. Queue가 비어있으면 대기(blocking)
4. 새로운 작업이 들어오면 다시 실행

즉, 메인 스레드는 아무 일도 하지 않는 것처럼 보여도
Looper가 계속 다음 작업을 기다리고 있기 때문에 종료되지 않는다.


개념적으로는 아래 코드와 비슷한 느낌
```kotlin
while (true) {
    val message = queue.next() // 없으면 여기서 blocking
    message.execute()
}
```

### ✅ Message Queue
> Message Queue는 처리할 작업들이 줄 서서 대기하는 공간이다.

- FIFO(First-In-First-Out) 구조로 동작한다.

#### Queue에 들어가는 대표 작업들

* 터치 이벤트
* 키보드 입력
* 화면 다시 그리기(invalidate)
* Activity 라이프사이클 콜백
* `Handler.post {}`
* Runnable 작업

예를 들어 사용자가 화면을 터치하면:

```text
터치 발생
   ↓
Message Queue에 이벤트 저장
   ↓
Looper가 이벤트 꺼냄
   ↓
Main Thread에서 처리
   ↓
UI 업데이트
```
--- 

## 💡 왜 Main Thread는 죽지 않을까?

* Main Thread 내부에는 Looper가 존재한다.
* Looper는 무한 루프를 돌며 Message Queue를 감시한다.
* Queue가 비어있으면 block 상태로 대기한다.
* 새로운 이벤트가 들어오면 즉시 처리한다.

결론:

> Main Thread를 계속 살아있게 만드는 존재 Looper 때문이다.
