## Bounded Service란?

**클라이언트(Activity, Fragment 등)가 `bindService()`로 연결되어 상호작용하는 Service.**

### 1. 특징
- 클라이언트가 `bindService()`로 연결하고, 연결된 동안 서비스의 메서드를 직접 호출할 수 있다.
- 서비스는 IBinder 객체를 반환하여 클라이언트와의 통신 채널을 제공한다.
- 여러 클라이언트가 하나의 서비스에 동시에 bind 가능하며, 시스템이 연결된 클라이언트 수를 관리한다.
- 모든 클라이언트가 `unbindService()` 하면 시스템이 자동으로 서비스를 종료시킨다. (별도 stop 호출 불필요)

### 2. 생명주기
> bindService() 호출 -> onCreate() -> onBind() -> 서비스 사용 중 -> unbindService() -> onUnbind() -> onDestroy() -> 종료
- `onCreate()` : 서비스 인스턴스 최초 생성 시 1회만 호출. (초기화 작업)
- `onBind()` : 클라이언트가 bindService() 호출 시 실행. IBinder 객체를 반드시 반환해야 함. (통신 채널 생성 진입점)
- `onUnbind()` : 모든 클라이언트가 unbind 했을 때 호출. true 반환 시 이후 재바인딩 시 onRebind() 호출됨.
- `onRebind()` : onUnbind()가 true를 반환한 상태에서 새 클라이언트가 bind 할 때 호출. (onBind() 대신 호출됨)
- `onDestroy()` : 모든 클라이언트가 unbind 한 뒤 호출. 리소스 해제 위치.

### 3. 주의 사항
- **메모리 누수 방지**: 클라이언트는 onStop() 등에서 반드시 unbindService() 호출 필요.
- **중복 unbind 금지**: 이미 unbind된 connection을 다시 unbind하면 IllegalArgumentException 발생. → bound 플래그로 상태 관리.
- **Started + Bound 조합 시**: startService()로도 시작했다면 모든 클라이언트가 unbind 해도 onDestroy()가 호출되지 않는다. 반드시 `stopSelf()` / `stopService()`도 호출해야 종료됨. (두 종료 조건을 모두 만족해야 함)

### 4. 통신 방식 3가지
1. **Binder 클래스 확장**: 같은 프로세스 (앱 내부). 가장 일반적이고 권장
2. **Messenger**: 다른 프로세스 IPC, 단일 스레드 메시지 큐로 충분할 때
3. **AIDL**: 다른 프로세스 IPC + 멀티스레드 동시 처리 필요 시
